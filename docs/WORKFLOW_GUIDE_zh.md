# 架构与工作流深度解析

本文档旨在详细阐述 **academic-auto-reviewer** 生态系统的内部运行机制。它面向进阶用户、开发者及研究人员，旨在针对学术初稿如何通过自动化、零幻觉的同行评审提供完全透明的解释。

---

## 核心理念：零幻觉智能代理 RAG

大型语言模型 (LLM) 因虚构学术引用和错误证实实证陈述而“闻名”。

为解决这一问题，我们的生态系统在 `auditor`（事实核查代理）上实施了严格的**自然语言推理 (NLI)** 约束。该代理被完全禁止调用其内部预训练记忆或使用实时网页搜索工具。它**仅能**根据本地抓取的准确文本摘录来评估陈述的真实性。

---

## 阶段 1：输入解析与命令触发

整个流程始于用户在 AI 代理终端中触发工作流：

```bash
/paper-review drafts/my_manuscript.md --voice third
```

一旦触发，`orchestrator`（调度器）会解析您的参数。如果检测到 `--voice third`，它将应用第三人称叙述偏好，以确保所有专业代理在语言风格上保持一致。

---

## 阶段 2：预处理管线 (`tools/`)

为了最大化 Token 利用率并降低 AI 编辑的“认知负荷”，调度器将清理过程委派给本地 Python 脚本。

### 1. `preprocess_manuscript.py`
该工具读取您的初稿并剔除与修辞分析无关的元素：
- **参考文献与书目区块**
- **图表、图像及其标题**
- **表格与数据结构**

输出为一个轻量级的 `_clean.md` 文件，仅包含纯文本叙述。该文件随后被干净地拆分为多个章节（如 `ch01.md`, `ch02.md`），以显著提高 LLM 对每个章节的处理精度和注意力跨度。

---

## 阶段 3：证据链接工作流

在 `auditor` 接收文档之前，系统会执行 `batch_factcheck_prep.py`。该脚本代表了检索增强生成 (RAG) 的关键链接。

1. **提取**：利用正则匹配提取初稿中所有符合 APA/作者-年份标准的引用。
2. **书目检索**：将引用与您的本地 `.bib` 文件进行交叉比对，找到对应的 `citekey`。
3. **数据库抓取**：识别出 `citekey` 后，执行 `read_literature.py` 从您的**本地 Markdown 数据库**（由 *mark-lit-down* 预先生成）中提取准确原文。
4. **JSON 构建**：将引用及其对应的准确源文本编译为一个高度约束的 JSON 文件。

---

## 阶段 4：代理并行调度

预处理和 JSON 构建完成后，`orchestrator` 将在并行执行空间启动三个专业代理。

### 语言轨道 (`linguist`)
逐句扫描拆分后的章节。
- 执行严格的语法规则（牛津逗号、中英文空格、标点嵌套）。
- 禁止改变语气、原意或句子结构。
- 输出一个可操作的日志，标明准确行号及修改前后的文本。

### 结构轨道 (`architect`)
评估章节的宏观逻辑流。
- 标记逻辑断层、冗余表达、过度对冲 (over-hedging) 以及衔接薄弱之处。
- 评估宏观结构约束（例如：*引言中提出的假设是否在方法论中得到了准确测试？*）。
- 输出包含修改建议的结构连贯性日志。

### 审计轨道 (`auditor`)
消耗阶段 3 生成的 JSON 上下文文件。
- 将手稿中的陈述与本地证据块进行并排对比。
- 将每个引用分类为：`SUPPORTED` (支持)、`CONTRADICTED` (矛盾)、`UNVERIFIABLE` (无法验证) 或 `NEUTRAL` (中立)。
- 输出一个 Markdown 表格，对您的文献综述完整性进行交叉验证。

---

## 阶段 5：结果汇总

当所有代理发出完成信号后，`orchestrator` 会收集所有日志和报告。您的原始初稿将保持原始状态，未做任何修改，但您现在已拥有：

- `[Proofreading Log]` (语言润色日志)
- `[Structural Flow Log]` (结构流日志)
- `[Fact-Check Validation Report]` (事实核查结果报告)

每一处修改建议都是透明且可操作的。您可以像处理顶级期刊编辑提供的反馈一样，根据这些建议进行精准迭代。
