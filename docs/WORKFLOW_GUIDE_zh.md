# 系统架构与工作流说明

本文档面向进阶用户、开发者及研究人员，深入说明 **academic-auto-reviewer** 的底层运行机制，旨在就学术初稿如何经由自动化同行评审流程得到处理，提供完整透明的技术说明。

---

## 核心设计理念：多智能体协作与无幻觉 RAG

本系统采用由 **Orchestrator（调度中心）** 统筹的多智能体并行工作流：由 **Linguist（语言专家）** 负责润色，**Architect（架构专家）** 负责逻辑评估，而 **Auditor（审计专家）** 则承担最关键的事实核查任务。

大型语言模型（LLM）由于过度依赖预训练权重，极易“虚构”学术引用或错误证实实证陈述。为了应对这一挑战，我们为 **Auditor** 智能体设定了严苛的**自然语言推理（NLI）**约束。该智能体被完全禁止调用内部记忆或实时搜索，仅能依据由 *mark-lit-down* 从本地知识库提取的精准文本片段，对陈述的真实性进行闭环判定。

---

## 阶段一：输入解析与命令触发

工作流由用户在智能体终端中主动触发：

```bash
/paper-review drafts/my_manuscript.md --voice third
```

触发后，`orchestrator`（调度器）解析命令参数。若检测到 `--voice third` 标志，系统将统一采用第三人称叙述风格，以确保各专业智能体的输出在语言层面保持一致。

---

## 阶段二：预处理管线（`tools/`）

为提高 token 利用效率并降低模型处理负担，调度器将前置清理任务交由本地脚本执行。

### `preprocess_manuscript.py`

读取初稿并移除非正文叙述元素，包括：

- 参考文献与书目列表
- 图表、图像及其说明文字
- 表格与结构化数据

处理后的文档将按逻辑章节切分（如 `ch01.md`、`ch02.md`），以提升模型对各章节内容的专注度与理解精度。

---

## 阶段三：自动化证据索引

进入审计流程前，系统执行核心预处理程序，构建"证据驱动"RAG 的索引基础：

1. **引用提取**：以正则表达式提取初稿中所有符合 APA 或作者-年份格式的引用标注。
2. **书目检索**：将提取的引用与本地 `.bib` 文件交叉比对，获取对应的 `citekey`。
3. **原文抓取**：识别 `citekey` 后，调用 `read_literature.py` 从本地 Markdown 文献数据库（由 *mark-lit-down* 预先生成）中提取相应原文。
4. **JSON 构建**：将引用与对应原文整合为结构化 JSON 文件，供后续智能体调用。

---

## 阶段四：智能体并行调度

预处理与证据索引完成后，`orchestrator` 并行启动三个专项智能体。

### 语言轨道（`linguist`）

逐句扫描各章节切片，执行语言层面的审查：

- 检查语法规范（含牛津逗号、中英文间距、标点嵌套等细节）。
- 不得改变原文语气、语义或句子结构。
- 输出可操作的校对日志，注明具体行号及修改前后的对比。

### 结构轨道（`architect`）

评估章节间的宏观逻辑连贯性：

- 标注逻辑断层、冗余表达、过度规避性措辞（over-hedging）及衔接薄弱处。
- 检验宏观结构约束，例如：*引言中提出的研究假设是否在方法论部分得到了对应的检验设计？*
- 输出包含逻辑诊断与修改建议的结构分析日志。

### 审计轨道（`auditor`）

挂载阶段三生成的证据索引 JSON，执行事实核查：

- 将手稿中的实证陈述与本地证据片段逐一对照。
- 对每处引用作出分类判定：`SUPPORTED`（有据可查）、`CONTRADICTED`（与文献相悖）、`UNVERIFIABLE`（无法核实）或 `NEUTRAL`（中性陈述）。
- 输出 Markdown 表格，呈现文献综述的完整性核查结果。

---

## 阶段五：结果汇总与成果交付

各智能体分析完成后，调度器汇总输出结果。原始稿件保持不变，同时生成三份独立的专项报告：

- `[Proofreading Log]` — 语言校对建议与排版规范核查明细
- `[Structural Flow Log]` — 论证逻辑分析与章节衔接评估
- `[Fact-Check Validation Report]` — 基于本地数据库的事实核查报告

所有审阅意见均具备可追溯性与可操作性，可作为专家预审意见参考，用于手稿的精准迭代修改。
